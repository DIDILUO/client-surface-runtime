# 统一内容接口契约

> 文件版本：v0002
> 更新时间：2026-03-25 17:46:00
> Encoding: UTF-8
> 本次更新：补充 settings 场景、service_entry 资源、service_action 动作与 profile_asset 资源口径

## 1. 目标

本契约用于统一全站内容接口，不按单个页面拆协议，而按“资源域 + 场景聚合”组织。

目标只有四个：

1. 让前端当前首页、后续活动页、个人中心页和各内页共用一套内容体系。
2. 让后端只维护一套统一内容域，不因页面增多而接口失控。
3. 让资源之间可以互相跳转，不重新发明新协议。
4. 让前端当前已做工作不报废，后续只需补 adapter 接入。

---

## 2. 核心原则

### 2.1 不按页面建接口体系

错误方式：

1. 首页一套协议。
2. 活动页一套协议。
3. 个人中心一套协议。
4. 每个内页再来一套协议。

正确方式：

1. 统一资源模型。
2. 少量场景聚合接口。
3. 统一详情接口。
4. 统一动作接口。

### 2.2 场景只负责聚合，资源才是真正主体

场景：

1. `home`
2. `activity`
3. `profile`
4. `settings`

资源：

1. `notice`
2. `activity`
3. `drop`
4. `market_item`
5. `category`
6. `asset`
7. `user_profile`
8. `service_entry`
9. `profile_asset`

### 2.3 页面只消费前端稳定模型

统一执行链路：

`后端 DTO -> 前端 adapter -> 前端 model -> 页面`

页面层不得：

1. 直接吃后端原始结构。
2. 直接拼字段兼容逻辑。
3. 直接绑定后端路径或后端内部命名。

---

## 3. 前后端职责划分

### 3.1 前端负责

前端必须负责的事情：

1. 页面路由组织。
2. `resourceType + resourceId` 驱动详情页。
3. DTO 到稳定模型的转译。
4. loading / empty / error 状态。
5. 页面交互状态与局部缓存。
6. 金额、数量、时间等展示格式化。
7. 场景聚合数据的页面消费。

前端不负责：

1. 内容真值存储。
2. 上下架决策。
3. 后台配置真值。
4. 图片派生图生成。
5. 公告已读持久化真值。

### 3.2 后端负责

后端必须负责的事情：

1. 内容资源存储。
2. 内容后台配置。
3. 场景聚合输出。
4. 详情按资源 ID 查询。
5. 公告已读写回。
6. 资源间关联关系维护。
7. 分类过滤、排序、分页。
8. 图片主图存储与派生图生成。

后端不负责：

1. 前端页面路径。
2. 前端组件结构。
3. 前端动画和状态样式。
4. 前端格式化展示细节。

### 3.3 双方共同约束

1. 后端不返回前端页面路由。
2. 前端不把运营配置写死成真值。
3. 后端不返回混乱 UI 字段名给页面直接吃。
4. 前端不按一资源一页面硬编码。

## 4. 统一内容域设计

建议后端只维护一个内容域：

`content`

当前建议的最小接口集合：

1. `GET /api/content/scene`
2. `GET /api/content/resource`
3. `GET /api/content/list`
4. `POST /api/content/action/notice-read`

当前首页立即需要的最小交付：

1. `GET /api/content/scene?sceneId=home`
2. `GET /api/content/resource?resourceType=notice&resourceId=...`
3. `GET /api/content/resource?resourceType=drop&resourceId=...`
4. `GET /api/content/resource?resourceType=market_item&resourceId=...`
5. `POST /api/content/action/notice-read`

当前已知独立页与服务入口需要补充的最小交付：

1. `GET /api/content/scene?sceneId=settings`
2. `GET /api/content/resource?resourceType=service_entry&resourceId=orders`
3. `GET /api/content/resource?resourceType=service_entry&resourceId=auth`
4. `GET /api/content/resource?resourceType=service_entry&resourceId=wallet`
5. `GET /api/content/resource?resourceType=service_entry&resourceId=invite`
6. `GET /api/content/resource?resourceType=service_entry&resourceId=community`
7. `GET /api/content/resource?resourceType=profile_asset&resourceId=M-01`

这样做的好处：

1. 后端不需要给每个页面单独发明接口。
2. 前端场景页首屏只打一枪。
3. 后续内页全部收敛到统一详情接口。
4. 资源互链天然统一。

## 5. 统一响应包格式

所有接口统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req_20260325_0001",
  "serverTime": "2026-03-25T21:45:00+08:00",
  "data": {}
}
```

字段要求：

1. `code`
   - `0` 表示成功。
   - 非 `0` 表示失败。
2. `message`
   - 简短说明。
3. `requestId`
   - 后端日志追踪。
4. `serverTime`
   - 服务端时间。
5. `data`
   - 真正业务内容。

## 6. 统一资源标识规则

所有资源统一使用：

1. `resourceType`
2. `resourceId`
3. `status`
4. `updatedAt`

资源类型枚举建议：

1. `notice`
2. `activity`
3. `drop`
4. `market_item`
5. `category`
6. `asset`
7. `user_profile`
8. `service_entry`
9. `profile_asset`

统一跳转引用协议：

```json
{
  "target": {
    "targetType": "drop",
    "targetId": "AX-99"
  }
}
```

要求：

1. 后端只返回资源引用。
2. 前端自己做页面路由映射。
3. 不允许后端返回 `pagePath` 这类前端实现字段。
4. 非资源详情动作统一挂到动作类型，例如：
   - `market_action`
   - `settings_action`
   - `service_action`

## 7. 场景聚合接口

### 7.1 接口定义

`GET /api/content/scene`

### 7.2 请求参数

1. `sceneId`
2. `platform` `（可选）`
3. `channel` `（可选）`
4. `locale` `（可选）`

当前场景值建议：

1. `home`
2. `activity`
3. `profile`
4. `settings`

### 7.3 返回结构总形态

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req_scene_0001",
  "serverTime": "2026-03-25T21:45:00+08:00",
  "data": {
    "sceneId": "home",
    "version": 1,
    "updatedAt": "2026-03-25T21:00:00+08:00",
    "blocks": []
  }
}
```

### 7.4 为什么用 `blocks`

因为后面不止首页。

如果直接把返回结构写死成：

1. `noticeBar`
2. `banners`
3. `featuredDrop`
4. `market`

那后续活动页、个人中心页会再次长歪。

所以统一建议：

1. 场景页返回 `blocks`
2. 每个区块有 `blockType`
3. 前端 adapter 按 `blockType` 转成页面模型

### 7.5 首页场景 block 示例

```json
{
  "sceneId": "home",
  "blocks": [
    {
      "blockType": "notice_bar",
      "label": "公告",
      "detailLabel": "查看公告详情",
      "items": []
    },
    {
      "blockType": "banner_carousel",
      "items": []
    },
    {
      "blockType": "featured_drop",
      "item": {}
    },
    {
      "blockType": "market_overview",
      "sectionTitle": "藏品市场",
      "sectionSubtitle": "MARKET FLOW",
      "categories": [],
      "actions": [],
      "items": []
    }
  ]
}
```

### 7.6 活动页场景 block 示例

```json
{
  "sceneId": "activity",
  "blocks": [
    {
      "blockType": "activity_banner",
      "items": []
    },
    {
      "blockType": "activity_notice_list",
      "items": []
    },
    {
      "blockType": "activity_feed",
      "items": []
    }
  ]
}
```

### 7.7 个人中心场景 block 示例

```json
{
  "sceneId": "profile",
  "blocks": [
    {
      "blockType": "profile_summary",
      "item": {}
    },
    {
      "blockType": "asset_overview",
      "items": []
    },
    {
      "blockType": "quick_actions",
      "items": []
    }
  ]
}
```

## 8. 统一资源详情接口

### 8.1 接口定义

`GET /api/content/resource`

### 8.2 请求参数

1. `resourceType`
2. `resourceId`

请求示例：

`GET /api/content/resource?resourceType=drop&resourceId=AX-99`

### 8.3 返回结构

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req_resource_0001",
  "serverTime": "2026-03-25T21:45:00+08:00",
  "data": {
    "resourceType": "drop",
    "resourceId": "AX-99",
    "title": "合成黎明",
    "subtitle": "FEATURED DROP",
    "status": "online",
    "updatedAt": "2026-03-25T21:00:00+08:00",
    "summary": "用于详情头部摘要",
    "asset": {},
    "payload": {},
    "relations": []
  }
}
```

### 8.4 设计要求

1. 所有详情页都走同一接口。
2. `payload` 内放资源专属字段。
3. `relations` 用于资源互联。

### 8.5 `relations` 示例

```json
[
  {
    "relationType": "related_notice",
    "target": {
      "targetType": "notice",
      "targetId": "HOME-N-001"
    }
  },
  {
    "relationType": "related_activity",
    "target": {
      "targetType": "activity",
      "targetId": "ACT-002"
    }
  }
]
```

这样活动页、公告页、藏品页、个人中心页之间都能互联，不用重写跳转协议。

## 12. 图片资源策略

### 12.1 结论

内容侧只维护一张主图，这个方向是对的。

但前端不能长期直接拿原始最大图到处缩。

正确方式应该是：

1. 后台维护一张主图。
2. 后端或图片服务生成派生规格图。
3. 前端按场景吃对应规格图。

### 12.2 统一图片资源结构

```json
{
  "assetId": "ASSET-AX99-COVER",
  "originalUrl": "https://cdn.example.com/original/ax99.jpg",
  "width": 1600,
  "height": 1600,
  "focalPoint": {
    "x": 0.5,
    "y": 0.5
  },
  "variants": {
    "thumb": "https://cdn.example.com/thumb/ax99.jpg",
    "card": "https://cdn.example.com/card/ax99.jpg",
    "banner": "https://cdn.example.com/banner/ax99.jpg",
    "detail": "https://cdn.example.com/detail/ax99.jpg"
  }
}
```

### 12.3 前端负责什么

1. 按场景选择规格。
2. 不直接请求原始图做首屏展示。
3. 在没有派生图时降级显示。

### 12.4 后端负责什么

1. 主图存储。
2. 派生图输出。
3. 焦点点位维护。
4. URL 下发。

### 12.5 为什么不能只发最大图

1. 首页首屏会慢。
2. 移动端解码压力高。
3. 带宽浪费严重。
4. 一些场景需要裁切策略，不是简单缩小。

## 13. 当前前端应该怎么留接口

### 13.1 前端现在应该负责落的东西

1. 把当前首页、后续活动页、个人中心页的场景 service 统一成“场景聚合 service”。
2. 把详情页全部统一成“按 `resourceType + resourceId` 拉详情”。
3. 把当前 `route meta` 逐步收口到：
   - `targetType`
   - `targetId`
4. 把页面继续保持为稳定 model 消费层。
5. 提前给图片资源留 `asset` 结构接口位。

### 13.2 前端现在不该做的东西

1. 不要为每个藏品做独立页面协议。
2. 不要继续往页面里硬编码内容。
3. 不要让页面直接吃后端 DTO。
4. 不要自己实现图片派生图服务。

### 13.3 后端现在应该负责落的东西

1. `scene` 聚合接口。
2. `resource` 详情接口。
3. `list` 列表接口。
4. `notice-read` 动作接口。
5. 统一资源 ID 体系。
6. 统一图片资源输出。

## 14. 当前项目的最优落地顺序

### 第 1 批：前端先定边界，后端最小接入

前端：

1. 统一 `targetType + targetId`
2. 保持页面只消费 model
3. 保留 adapter 接口位

后端：

1. `GET /api/content/scene?sceneId=home`
2. `GET /api/content/resource`
3. `POST /api/content/action/notice-read`

### 第 2 批：扩到活动页、个人中心页

后端：

1. `GET /api/content/scene?sceneId=activity`
2. `GET /api/content/scene?sceneId=profile`
3. `GET /api/content/list`

前端：

1. 活动页改为场景聚合消费
2. 个人中心改为场景聚合消费
3. 各详情页统一走 `resource` 接口

### 第 3 批：图片和后台配置收口

后端：

1. 资源主图管理
2. 派生图输出
3. 后台配置接入内容域

前端：

1. 切换到 `asset.variants`
2. 不再依赖本地假图

## 15. 源码交付与接口文档边界

源码交付时：

1. 规范体系不进交付包。
2. 台账体系不进交付包。
3. 备份目录不进交付包。
4. 构建产物不进交付包。
5. 本契约文档必须进入交付包。
6. 对外交付的首页审核说明应放在本契约旁边。

原因：

1. 规范和台账属于内部治理资产，不属于源码交付物。
2. 接口契约属于实际协作资产，必须交付。
3. 面向外部审核的首页问题、修复和需求对齐结论也属于实际协作资产，应随源码一并交付。

配套白名单见：

1. `源码交付白名单.md`

配套审核说明见：

1. `首页外部审核说明.md`

## 16. 最终结论

最优方案就是：

1. 不按页面建接口。
2. 按资源域建接口。
3. 用场景聚合接口解决首页、活动页、个人中心首屏。
4. 用统一详情接口解决各内页。
5. 用统一列表接口解决市场、公告、活动、资产列表。
6. 个人资产继续走统一 `profile_asset` 资源，不再额外发明页面专属协议。
7. 用统一动作接口解决公告已读等行为。
8. 图片只维护一张主图，但前端最终吃派生规格图，不长期吃最大原图。

这套口径下：

1. 前端当前工作不会废。
2. 后端实现路径最短。
3. 后续页面再多也不会重新发明协议。
4. 资源互联天然统一。

## 18. 2026-03-26 焦点裁切补充口径

1. `asset.focalPoint.x` 与 `asset.focalPoint.y` 统一使用 `0~1` 的归一化坐标。
2. 后端不得下发 `0~100` 百分比语义；前端不会兼容双语义输入。
