## 3. 已完成修复

### 3.1 首页壳层与入口来源

已完成：

1. 移除首页 `mode-web` 双壳层脚本分支。
2. 首页统一为单壳层响应式主路径。
3. 抽屉入口与个人中心金刚区入口来源拆分正确。
4. 抽屉延迟跳转增加定时器句柄、卸载清理和组件存活保护。

### 3.2 活动页与个人中心内容链

已完成：

1. 活动页内容加载改为 `try/catch/finally`。
2. 个人中心内容加载改为 `try/catch/finally`。
3. 两页失败时都会回落到 shell 内容，不再抛未处理 rejection。

### 3.3 Banner 焦点裁切与边界保护

已完成：

1. `focalPoint` 契约已明确为 `0~1` 归一化坐标。
2. 页面层已对 `focalPoint` 做 `clamp`。
3. 非法值、缺失值、NaN 均会回落到安全值。

## 4. 当前明确需求

首页 UI 必须适用于以下场景：

1. APP 打包壳
2. 手机 Web
3. 电脑 Web

并且上述场景下需覆盖：

1. Android
2. iOS
3. 鸿蒙

## 5. 需求复核结论

基于上面的正式需求，当前首页修复状态应这样理解：

### 5.1 已满足需求的项

以下问题已真正收口：

1. 首页双壳层脚本分支
2. 抽屉 `window.setTimeout` 依赖
3. 抽屉延迟跳转未清理句柄
4. 个人中心金刚区入口来源失真
5. 活动页/个人中心页未处理 rejection
6. `focalPoint` 缺少输入边界保护

### 5.2 尚未最终满足需求的项

以下问题在“当前代码安全性”层面已处理，但在“多端最终能力”层面仍未完全满足需求：

1. banner 焦点裁切的跨端一致性

现状：

1. H5 端：按 `focalPoint` 精确裁切。
2. 非 H5 端：当前降级为中心裁切安全路径。

这意味着：

1. 当前代码已经避免了异常值把 banner 裁飞。
2. 但如果以“APP 打包壳 + 手机 Web + 电脑 Web + Android/iOS/鸿蒙”作为正式验收标准，
3. 那么“非 H5 端仅中心裁切降级”仍然只是过渡方案，不是最终收口。

## 6. 对外审核建议口径

外部审核时建议直接按以下结论使用：

1. `focalPoint` 边界保护：已修复。
2. `Drawer` 延迟跳转清理：已修复。
3. banner 非 H5 焦点裁切：当前不是遗漏 bug，而是已显式声明的降级实现。
4. 但如果按全端一致体验验收，该项仍应继续开批次补齐。

## 7. 当前结论

首页当前状态分两层：

1. 代码安全性与工程稳定性：本轮已收口。
2. 跨端一致体验：banner 非 H5 焦点裁切仍需后续增强批次。

因此当前最准确的结论是：

1. 首页大部分已知问题已修复。
2. banner 多端焦点裁切不是“未修 bug”，但也不是“最终满足全端需求”。
